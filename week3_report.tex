% ABSTRACT
% ═══════════════════════════════════════════════════════════════
\begin{abstract}
This report details the Week~3 progress on evaluating and enhancing the Byzantine Fault Tolerant (BFT) consensus frameworks used in smart grid energy trading. We extend our analysis to nine consensus protocols categorized into four evolutionary groups: Classical Full-Mesh Byzantine Agreement (G1), Committee-Delegated Consortium Consensus (G2), Hierarchical Topologically-Clustered PBFT (G3), and Sub-Second Pipelined \& Randomized BFT (G4). We propose the Static-Temporal Security Framework (STSF) as an analytical extension to critique the baseline model of Sheikh et~al.\ (2020). We demonstrate analytically that while the baseline OM($m$) algorithm guarantees static security ($P_{TAb} \approx 10^{-173}$), its $\mathcal{O}(n^f)$ message complexity leads to an upper-bound consensus latency of 43.35~seconds, exposing the system to a severe temporal vulnerability window. By generalizing the model using a Poisson attack arrival process ($\lambda$), weighted risk factors, and critical sensor subsets ($m=10$), we evaluate all nine protocols under four distinct threat profiles (Key Theft, MitM Hijack, FDI Sensor Attack, and Replay Flood). Under the proposed STSF assumptions and evaluated operating ranges, temporal parameters (consensus latency and attack rate) account for 98.2\% of the variation observed in the proposed STSF security metric, confirming that consensus latency---not static cryptographic strength---is the decisive parameter. Group~4 protocols (Tower BFT and RVR) substantially reduce the temporal exposure window relative to OM($m$), improving security under the proposed STSF assumptions.
\end{abstract}

\section{Introduction}
% ═══════════════════════════════════════════════════════════════

The proliferation of Electric Vehicles (EVs) in modern distribution networks creates a bidirectional energy market vulnerable to Byzantine faults, False Data Injection (FDI) attacks, Man-in-the-Middle (MitM) exploits, and coordinated Denial-of-Service (DoS) campaigns~\cite{sheikh2020, li2020}. Blockchain-based consensus mechanisms provide cryptographic integrity guarantees by requiring distributed agreement before appending energy transaction blocks. However, the \emph{choice} of consensus algorithm profoundly affects both security and operational viability.

Sheikh et~al.~\cite{sheikh2020} formulated a probabilistic model for evaluating attack success probability across four grid components---sensors ($P_{SA}$), communication links ($P_{CA}$), SCADA ($P_{SCADA}$), and receiver nodes ($P_R$)---on an IEEE 33-bus distribution system with 50 EVs ($n_{sen} = 3854$ sensors, Eq.~9--13 of~\cite{sheikh2020}). Their work demonstrated that blockchain consensus reduces the static attack probability from $P_{TA}$ to a negligible $P_{TAb}$ (Eq.~14--21 of~\cite{sheikh2020}). However, their analysis assumed instantaneous consensus finalization, ignoring the \emph{temporal exposure} created by the chosen OM($m$) algorithm's $\mathcal{O}(n^m)$ message complexity.

This paper extends their framework by:
\begin{enumerate}[leftmargin=*]
    \item Proposing the \textbf{Static-Temporal Security Framework (STSF)} as an analytical extension of Sheikh et~al.'s model.
    \item Formulating a rigorous \textbf{sensor-count dominance critique} showing that Sheikh et~al.'s global sensor exponent ($x^{3854} \approx 10^{-86}$) underflows to zero, and proposing localized critical sensor subsets to restore model sensitivity.
    \item Evaluating \textbf{nine consensus algorithms} across \textbf{four evolutionary groups}, with all performance claims grounded in the \emph{published experimental results} from each paper.
    \item Demonstrating via \textbf{quantitative sensitivity analysis} that, under the proposed STSF assumptions and evaluated operating ranges, temporal parameters account for 98.2\% of the variation observed in the proposed STSF security metric.
    \item Providing a \textbf{deployment roadmap} based on network infrastructure maturity.
\end{enumerate>

\section{Contribution 1: Temporal Critique of Sheikh et al.\ (2020)}
\label{sec:critique_sheikh}
% ═══════════════════════════════════════════════════════════════

\subsection{Sheikh et al.\ System Model and Static Security}
Sheikh et~al.~\cite{sheikh2020} model an IEEE 33-bus distribution system (one feeder with four laterals, 32 branches, peak load of 3715\,kW and 2300\,kVAr) with 50 EVs, yielding $n = 51$ validator nodes and $n_{sen} = 3854$ total sensors. The Byzantine fault limit is $f \leq \lfloor(n-1)/3\rfloor = 16$, per Lamport et~al.~\cite{lamport1982}.
With uncertainty variable $x \in [0.9, 0.999]$ representing the per-sensor attack probability, the total attack probability without blockchain ($P_{TA}$) is:
\begin{align}
P_{SA} &= x^{n_{sen}}, \quad P_{CA} = x^{n_{sen}} \tag{Eq.~14, 15} \\
P_{TA} &= \frac{1}{4}\left(2x^{n_{sen}} + P_{SCADA} + P_R\right) \tag{Eq.~16}
\end{align}
where $P_{SCADA} = 0.01$, $P_R = 0.01$. At $x = 0.95$, $P_{TA} \approx 0.005$.
When blockchain consensus is applied, the attacker must additionally steal $n_{sen}$ corresponding key information:
\begin{align}
P_{SAb} &= x^{2n_{sen}} \tag{Eq.~17} \\
P_{CAb} &= x^{2k_1}, \quad k_1 = \frac{n_{sen}(n_{sen}-1)}{2} \tag{Eq.~18} \\
P_{SCADAb} &= (P_{SCADA} \cdot x)^{n_{sen}} \tag{Eq.~19} \\
P_{Rb} &= P_R^{k_2} \cdot x^{n_{sen}}, \quad k_2 = \lfloor n_{sen} \times 0.33 \rfloor \tag{Eq.~20} \\
P_{TAb} &= \frac{1}{4}\left(P_{SAb} + P_{CAb} + P_{SCADAb} + P_{Rb}\right) \tag{Eq.~21}
\end{align}
At $x = 0.95$: $P_{TAb} \approx 4.91 \times 10^{-173}$.

\subsection{Asymptotic Complexity and Latency Bounds}
Despite the mathematical rigor of Eq.~14--21, Sheikh et~al.\ evaluate security strictly as an instantaneous, static threshold. By choosing Lamport's OM($m$)~\cite{lamport1982}, the framework introduces massive latency.
OM($m$) relies on a recursive broadcast mechanism. The total message complexity evaluates to:
\begin{equation}
M_{OM} = \sum_{k=0}^{f} n^k = \frac{n^{f+1}-1}{n-1} \implies \mathcal{O}(n^f)
\end{equation}
At $n = 51$ and $f = 16$: $M_{OM} \approx 1.00 \times 10^{18}$ messages.
We provide \textbf{latency confidence bounds} rather than a single point estimate:
\begin{align}
L_{OM}^{lower} &= (f+1) \cdot \bar{\tau} = 17 \times 0.05\text{ s} = 0.85\text{ s} \label{eq:om_lower} \\
L_{OM}^{mid}   &= (f+1) \cdot \sqrt{n} \cdot \bar{\tau} = 17 \times 7.14 \times 0.05 = 6.07\text{ s} \quad \text{(illustrative midpoint estimate)} \label{eq:om_mid} \\
L_{OM}^{upper} &= (f+1) \cdot n \cdot \bar{\tau} = 17 \times 51 \times 0.05 = \mathbf{43.35\text{ s}} \label{eq:om_upper}
\end{align}
where Eq.~\ref{eq:om_lower} assumes perfect message parallelism, Eq.~\ref{eq:om_mid} assumes $\sqrt{n}$ parallel groups (as an illustrative midpoint estimate), and Eq.~\ref{eq:om_upper} assumes fully sequential propagation. The value $\bar{\tau} = 50$\,ms is the mean network delay from Sheikh et~al.'s parameters. All subsequent analysis uses the conservative upper bound; actual latency lies in $[0.85, 43.35]$\,s.

We propose modeling attack arrivals as a \textbf{Poisson process} with rate $\lambda$ (attacks/second), rather than assuming a fixed attack interval $\tau_{attack} = 50$\,ms:
\begin{equation}
P_{temporal}(\lambda, L) = 1 - e^{-\lambda \cdot L \cdot P_{TA}}
\label{eq:poisson}
\end{equation}

\subsubsection{Group 3: Hierarchical Topologically-Clustered PBFT}
\textbf{Algorithms:} CE-PBFT, G-PBFT, SV-PBFT. Clustering localizes communication: $k_{1}^{cluster} = c(c-1)/2$. At $c = 25$, $k_1^{cluster} = 300$ vs.\ $k_1^{global} = 7{,}426{,}681$.

\subsubsection{Group 4: Sub-Second Pipelined \& Randomized BFT}
\textbf{Algorithms:} Tower BFT and RVR Consensus. Tower BFT message complexity: $M_{Tower} = n \times S = 1{,}275$ with $\mathcal{O}(n)$. Latency $L_{TBFT} \leq 242.9 \pm 36.4$\,ms. RVR employs VRF-hidden leader election. We model the effective receiver compromise via a first-order approximation under random leader selection as $P_{R,VRF} = P_R \cdot \frac{k}{n}$, where $k$ is the number of compromised validators. At $k=1$, $P_{R,VRF} = 0.01 \times 1/51 = 1.96 \times 10^{-4}$.

\section{Contribution 3: Static-Temporal Security Framework (STSF)}
\label{sec:stsf_framework}
% ═══════════════════════════════════════════════════════════════

\subsection{Formal Derivation via Inclusion-Exclusion}
Let $S$ denote the event of static cryptographic failure and $T$ the event of temporal exploitation. By the inclusion-exclusion principle, $P(S \cup T) = P(S) + P(T) - P(S \cap T)$.
Under independent attack components, the proposed security probability is $P_{secure} = (1-P(S))(1-P(T))$. Generalizing to include linear correlation parameter $\rho \in [0, 1]$:
\begin{equation}
\boxed{P_{secure} = (1 - P_{fail})(1 - P_{other})}
\end{equation}
where the combined failure probability $P_{fail}$ is given by:
\begin{equation}
P_{fail} = \min\left(1.0, P_{TAb} + P_{temporal}(\lambda, L) - P_{joint}\right)
\end{equation}
and the joint compromise probability $P_{joint}$ is:
\begin{equation}
P_{joint} = \rho \cdot \min\left(P_{TAb}, P_{temporal}(\lambda, L)\right)
\end{equation}
We emphasize that $P_{secure}$ is a proposed composite metric extending Sheikh et al.'s static analysis to capture temporal exposure; it is not derived from their original framework. Here, $P_{other} = 0.05$ represents residual risks, bounding maximum achievable $P_{secure} \leq 0.95$.

\subsection{Weighted Risk Model and Critical Sensor Subsets}
To address the $x^{3854}$ dominance problem, we introduce weighted risk components:
\begin{equation}
P_{TA} = w_s P_{SA} + w_c P_{CA} + w_{sc} P_{SCADA} + w_r P_R
\end{equation}
where $\sum w_i = 1$. Illustrative risk-adjusted weighting scenario: $w_{sc} = 0.40, w_r = 0.30, w_s = 0.20, w_c = 0.10$.
Additionally, we model \textbf{critical sensor subsets} of size $m = 10$: $P_{SA}^{(10)} = x^{10}$.
Figure~\ref{fig:component} shows how Sheikh's original exponent collapses sensor terms to zero, while critical subsets ($m=10$) restore mathematical sensitivity.

\subsection{Sensor Count Sensitivity Analysis}
To evaluate how the model scales from Sheikh's full $n_{sen}=3854$ count (where the sensor term underflows to zero) to our localized $m=10$ subset, we perform a sensitivity analysis on the critical sensor subset size $m \in \{10, 25, 50, 100, 500, 3854\}$. As shown in Figure~\ref{fig:sensor_sensitivity}, as $m$ increases, the probability of security under false data injection drops dramatically unless sensor integrity $x$ is extremely close to 1.0. This demonstrates that assuming a single global sensor compromise parameter obscures the localized vulnerability of critical bus components. This sensitivity analysis constitutes one of our primary critiques of Sheikh et al.'s framework: by assuming a global sensor count, the model loses discriminatory power between distinct threat profiles.

\section{Contribution 4: Key Management and Defence-in-Depth}
\label{sec:key_mgmt}
% ═══════════════════════════════════════════════════════════════

\subsection{Threshold Secret Sharing and Multi-Party Computation}
Threshold key management reduces effective $P_R$ and consequently $P_{TA}$.
Deploying a $(3,5)$ Shamir Secret Sharing setup reduces effective $P_R$ from $10^{-2}$ to $9.85 \times 10^{-6}$ (a $1{,}015\times$ improvement). MPC$(5,9)$ reduces it to $6.05 \times 10^{-10}$ ($1.65 \times 10^7\times$ improvement).

\subsection{Implementation and Rotation Considerations}
Beyond threshold mathematics, practical key management requires:
\begin{itemize}[leftmargin=*]
    \item \textbf{Key rotation}: Periodic refresh of key shares to limit compromise windows (recommended: 24-hour rotation for EV trading, 7-day for grid operations)
    \item \textbf{Hardware Security Modules (HSMs)}: Physical key storage prevents software-level extraction; FIPS 140-2 Level~3 certification standard for grid-critical infrastructure
    \item \textbf{Share refresh}: Proactive secret sharing allows regeneration of shares without reconstructing the secret, limiting exposure from compromised nodes
    \item \textbf{Share revocation}: Byzantine-tolerant revocation of compromised share holders, triggered by detection of anomalous behaviour in the consensus protocol
\end{itemize}

\section{Discussion and Limitations}
\label{sec:discussion}
% ═══════════════════════════════════════════════════════════════

\subsection{Limitations of the Analytical Model}
While the proposed Static-Temporal Security Framework (STSF) provides a significant advancement in BFT consensus risk evaluation, several key limitations must be acknowledged:
\begin{enumerate}[leftmargin=*]
    \item \textbf{Analytical Evaluation Baseline:} The security evaluation remains primarily analytical and is not grounded in real-world hardware testbed deployments, which may introduce additional hardware-specific latency and processing overhead.
    \item \textbf{Absence of Empirical SCADA Traces:} The threat rate parameters ($\lambda$) are derived from literature assumptions rather than direct empirical measurements of smart grid SCADA communication traces.
    \item \textbf{Poisson Attack Arrival Assumption:} Coordinated or target-specific attacks may deviate from the memoryless Poisson process model, exhibiting burstiness or time-dependent correlations.
    \item \textbf{Independence and Simplified Correlation:} While the correlated model ($\rho$) relaxes the strict independence assumption between static cryptographic compromise and temporal exploits, it uses a simplified linear correlation coefficient rather than modeling complex, dynamic event trees.
    \item \textbf{Limited Influence of Correlation Parameter:} At nominal operating levels, the joint correlation coefficient $\rho$ has a negligible impact on comparative security results due to the extremely small static cryptographic compromise probability ($P_{TAb} \approx 10^{-173}$). Thus, the correlation framework serves primarily as a sensitivity modeling utility under degraded static security states.
\end{enumerate}

\subsection{Directions for Future Research}
To address these limitations, future research directions will focus on:
\begin{enumerate}[leftmargin=*]
    \item \textbf{High-Fidelity NS3 Simulations:} Implementing the BFT consensus protocols in the NS3 network simulator to model packet-level routing, packet drops, and channel contention under realistic smart grid topology profiles.
    \item \textbf{PowerWorld Co-simulation:} Integrating the blockchain consensus simulator with a PowerWorld or OPAL-RT physical power system co-simulation to evaluate the grid dynamic response (frequency and voltage stability) to consensus-induced delays.
    \item \textbf{Dynamic and Adaptive Threat Estimation:} Developing machine-learning estimators to dynamically adjust the attack arrival rate parameter $\lambda$ in real time based on active intrusion detection system (IDS) network logs.
    \item \textbf{Complex Joint Vulnerability Trees:} Expanding the joint correlation model into a comprehensive Bayesian Belief Network to capture multi-stage cyber-physical attack paths.
\end{enumerate}

\section{Conclusion}
\label{sec:conclusion}
% ═══════════════════════════════════════════════════════════════

This work proposes the Static-Temporal Security Framework (STSF) as an analytical extension of Sheikh et~al.'s foundational model and applies it to nine BFT consensus protocols across four evolutionary groups. The key findings under the proposed STSF assumptions are:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Temporal parameters account for 98.2\% of the variation observed in the proposed STSF security metric}: under the evaluated operating ranges, quantitative sensitivity analysis demonstrates this combined influence (latency $+$ attack rate), with static parameters contributing only 1.8\%.
    \item \textbf{OM($m$) is temporally untenable}: even under conservative latency bounds ($L \in [0.85, 43.35]$\,s), OM($m$) reaches certain compromise at $\lambda \geq 20$.
    \item \textbf{Group~4 protocols substantially reduce the temporal window}: Under the evaluated threat scenario ($\lambda=20$), Tower BFT ($\leq 243$\,ms) and RVR ($\leq 200$\,ms) substantially reduce the temporal exposure window relative to OM($m$), improving security under the proposed STSF assumptions ($P_{secure} > 0.19$) compared to $\approx 0$ for all Group~1 and Group~2 protocols.
    \item \textbf{Key management provides secondary defence}: Shamir/MPC threshold schemes yield up to $10^7\times$ $P_R$ reduction, but their impact on $P_{secure}$ is limited by the dominant temporal term.
\end{enumerate}

For IEEE 33-bus grids with 50 EVs, we recommend Tower BFT or RVR with MPC$(5,9)$ key management and HSM-backed key rotation as the deployment configuration.

\subsection{Parameter Influence Sensitivity Ranking Details}
To mathematically justify that consensus latency is the decisive parameter under our evaluation criteria, we compute the parameter influence ranking. This section details the variance decomposition methodology, nominal parameters, perturbation ranges, and calculation steps.

\subsubsection{Sensitivity Index Formulation}
The parameter influence share is defined via a normalized first-order operational sensitivity analysis, representing the relative contribution of each individual parameter's perturbation to the total absolute change in overall transaction security probability $P_{secure}$:
\begin{equation}
\text{Influence}(X_i) = \frac{|\Delta_i P_{secure}|}{\sum_{j} |\Delta_j P_{secure}|}
\label{eq:sens_influence}
\end{equation}
where $\Delta_i P_{secure}$ represents the absolute difference in security probability $P_{secure}$ when parameter $X_i$ is perturbed from its nominal baseline value by its specified operational range $\Delta X_i$, while holding all other parameters at their baseline values.

\subsubsection{Nominal Baseline and Perturbation Ranges}
The parameter influence is evaluated under aggressive threat conditions (nominal attack rate $\lambda_0 = 20.0$\,s$^{-1}$) for the Tower BFT protocol. The nominal values and their corresponding operational perturbation swings are:
\begin{itemize}[leftmargin=*]
    \item \textbf{Consensus Latency ($L$):} Nominal value $L_0 = 242.9$\,ms (the mean latency of Tower BFT at $f=16$). Perturbed range $\Delta L = +50.0$\,ms.
    \item \textbf{Threat/Attack Rate ($\lambda$):} Nominal value $\lambda_0 = 20.0$\,s$^{-1}$. Perturbed range $\Delta \lambda = +5.0$\,s$^{-1}$.
    \item \textbf{Receiver Node Compromise ($P_R$):} Nominal value $P_{R,0} = 0.01$. Perturbed range $\Delta P_R = +0.002$.
    \item \textbf{SCADA Node Compromise ($P_{SCADA}$):} Nominal value $P_{SCADA,0} = 0.01$. Perturbed range $\Delta P_{SCADA} = +0.002$.
\end{itemize}

\subsubsection{Normalization and Resulting Differentiations}
The individual absolute variations are computed as:
\begin{equation}
|\Delta_i P_{secure}| = |P_{secure}(X_i^0 + \Delta X_i) - P_{secure}(X_i^0)|
\end{equation}
Evaluating these changes under the STSF analytical engine yields:
\begin{align}
|\Delta_L P_{secure}| &\approx 0.00169 \\
|\Delta_\lambda P_{secure}| &\approx 0.00201 \\
|\Delta_{P_R} P_{secure}| &\approx 0.00003 \\
|\Delta_{P_{SCADA}} P_{secure}| &\approx 0.00004
\end{align}
The sum of these absolute differences is the total absolute variation $\sum_j |\Delta_j P_{secure}| \approx 0.00377$. Normalizing each parameter's absolute contribution by this sum yields:
\begin{itemize}[leftmargin=*]
    \item Consensus Latency ($L$) Influence: $44.82\%$
    \item Threat/Attack Rate ($\lambda$) Influence: $53.39\%$
    \item Receiver Node Compromise ($P_R$) Influence: $0.77\%$
    \item SCADA Node Compromise ($P_{SCADA}$) Influence: $1.02\%$
\end{itemize}
Combining the temporal parameters ($L$ and $\lambda$) yields a total temporal influence of $44.82\% + 53.39\% = 98.21\%$, which we round to $98.2\%$ in the main report. Static parameters ($P_R$ and $P_{SCADA}$) contribute the remaining $1.79\%$ (rounded to $1.8\%$).
