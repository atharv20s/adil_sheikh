736: We acknowledge that the joint compromise probability $P_{\mathrm{Compromise}} = 1 - \prod_{j} (1 - P_j)$ assumes that all 12 attack vectors act as independent, parallel failure modes. In dynamic industrial networks, cyber-physical attacks are often correlated; for example, a successful SCADA breach significantly facilitates key compromise and actuator overrides. Our independent failure model represents a conservative, structural upper-bound failure limit for comparison. Future research will employ Bayesian Belief Networks (BBN) and Dynamic Markov Decision Processes (MDP) to model sequential state transition probabilities and conditional failure cascades across physical smart meters, consensus protocols, and RTU actuators.lockchain} are sound and correctly capture the probability bounds.

752: \section{Temporal Vulnerability}
753: \label{sec:temporal_benchmarking}
754: 
755: \emph{Having established that blockchain improves static security by $\sim$170 orders of magnitude (Part~I), we now evaluate how the temporal exposure window of different consensus algorithms affects the realized security gains.}
756: 
757: \subsection{Temporal Exposure Window}
758: 
759: Although blockchain provides static security, transactions remain unfinalized during the consensus process. This latency window ($L$) exposes the network to temporal attacks. We model this exposure using a Poisson attack arrival process with rate $\lambda$ (attacks/second):
760: \begin{equation}
761: P_{temporal}(\lambda, L) = 1 - e^{-\lambda \cdot L \cdot P_{TA}}
762: \end{equation}
763: 
764: The overall security probability ($P_{secure}$) incorporates this temporal risk:
765: \begin{equation}
766: P_{secure} = (1 - P_{TAb})(1 - P_{temporal}(\lambda, L))(1 - P_{other})
767: \end{equation}
768: where $P_{other} = 0.05$ represents residual risks.
769: 
770: 
771: The following protocol comparison assumes blockchain has already been selected. The objective is no longer to determine whether blockchain should be deployed, but rather which consensus protocol provides the best operational trade-off.
772: 
773: \section{Protocol Comparisons}
774: 
775: To operationalize these temporal findings, we evaluate the protocols across distinct architectural families.
776: 
777: \subsection{Classical BFT (OM($m$) and PBFT)}
778: Classical protocols provide foundational Byzantine tolerance but struggle with message complexity. As shown in Table~\ref{tab:complexity}, OM($m$) requires $\mathcal{O}(n^f)$ messages, totaling $\approx 2.14 \times 10^{27}$ messages for a 51-node network. This makes it physically impossible to deploy for real-time grid control. Classic PBFT improves this to $\mathcal{O}(n^2)$ (7,854 messages), allowing practical implementation but capping scalability. Due to high latency ($L = 7,650$\,ms), PBFT preserves only 44.2\% of the blockchain's static security ($P_{secure} = 0.442$).

882: \subsection{Shortlisted Architectural Security and Performance Features}
883: 
884: We focus on four features that directly influence system metrics:
885: 
886: \begin{enumerate}[leftmargin=*]
887:     \item \textbf{Proof of History (PoH):} VDF-based clock. Eliminates synchronous communication rounds, reducing message overhead from $\mathcal{O}(n^2)$ to $\mathcal{O}(n)$, yielding sub-second latency ($L=243$\,ms) and restoring $P_{secure}$ to 0.927~\cite{shoup2023}. 
888: \textbf{Tower BFT Analytical Latency Derivation:} The 242.9\,ms latency evaluated in Table~\ref{tab:security} represents an empirical measurement. Analytically, Tower BFT latency $L$ is bounded by the network flight time ($\tau=50$\,ms), PoH tick verification ($\sim42.9$\,ms), and turbine tree propagation ($\sim37.1$\,ms for $n=51$). Thus, the theoretical lower bound is $L_{base} = 2\tau + L_{PoH} + L_{turbine} \approx 180$\,ms. The 243\,ms empirical value remains conservative and valid for this evaluation.
889: 
890:     \item \textbf{VRF Leader Election:} Random verifiable leader rotation. Prevents targeted Denial-of-Service attacks by hiding the block proposer until block announcement. Reduces effective receiver compromise probability from $P_R=0.01$ to $P_{R,VRF}=P_R/n \approx 1.96 \times 10^{-4}$~\cite{wang2025}.
891:     \item \textbf{Geohash Geographic Clustering:} Reduces communication complexity from global validator network size $n$ to localized cluster size $c$. Reduces PBFT message counts from 7,854 to 1,900~\cite{xu2025}.
892:     \item \textbf{Credit-Evaluation Scoring:} Dynamically adjusts validator weights based on behavior history, reducing the threat probability of malicious validators ($p_c$)~\cite{ding2024}.
893: \end{enumerate}
