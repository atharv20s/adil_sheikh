% ABSTRACT
% ═══════════════════════════════════════════════════════════════
\begin{abstract}
This paper quantifies the cybersecurity improvement that Byzantine Fault Tolerant (BFT) blockchain consensus mechanisms introduce to Electric Vehicle (EV) energy trading networks under the \emph{Static–Temporal Security Framework (STSF)}. Moving beyond simple protocol benchmarking, we focus on the central question: \emph{How much does blockchain improve the security of an EV Energy Trading System compared to the same system without blockchain?} Using the IEEE 33-bus distribution system with 50 EVs and 3,854 physical sensors as our reference, we evaluate security improvements using Sheikh et al.'s foundational four-component probabilistic attack model, demonstrating that under baseline probabilistic model assumptions, blockchain reduces total attack probability from $P_{\text{TA}} \approx 0.005$ to $P_{\text{TAb}} \approx 10^{-173}$---a static security gain of approximately 170 orders of magnitude under the Sheikh formulation. We complement this with a probabilistic evaluation across 12 distinct cyber-physical attack vectors (including Sybil, Byzantine node, and DDoS attacks) to illustrate the breadth of defensive coverage. We then prove that consensus validation latency creates a temporal vulnerability window ($P_{\text{temporal}}$) governed by a Poisson arrival process. Pipelined sub-second protocols (e.g., Tower BFT, RVR) preserve over 92\% of static security gains ($P_{\text{secure}} \ge 0.927$), whereas classical consensus protocols (e.g., $OM(m)$) lose up to 99\% of security benefits due to latency exposure ($P_{\text{secure}} = 0.012$). Finally, we corroborate our analytical formulations across $10^6$ Monte Carlo simulation trials to within 0.5\% error with 95\% confidence intervals.
\end{abstract}

 In particular, peer-to-peer (P2P) energy trading between Electric Vehicles (EVs) and Advanced Metering Infrastructure (AMI) provides grid balancing and ancillary services. However, this decentralized structure exposes the system to cyber-physical threats, including False Data Injection (FDI), Man-in-the-Middle (MitM), and Denial-of-Service (DoS) attacks.

While blockchain consensus has been proposed to secure these networks, existing literature often treats security as an instantaneous, static property or focuses solely on consensus performance benchmarking. This paper addresses a more fundamental research question:

\begin{quote}
\textbf{``How much does blockchain improve the security of an EV Energy Trading System compared to the same system without blockchain, and how does the choice of consensus algorithm affect this difference?''}
\end{quote}

To address this, we structure the evaluation into two parts. In the first part, we model the system without blockchain, derive the success probabilities for 12 core cyber-attacks, and compare them directly against a blockchain-protected baseline. In the second part, we evaluate how the temporal exposure window of different consensus algorithms affects the realized security gains, using throughput, latency, communication cost, and scalability.

Specifically, relative to the baseline study by Sheikh et al.~\cite{sheikh2020}, the core contributions proposed and evaluated in this paper are:
\begin{itemize}[leftmargin=*]
    \item \textbf{Localized Sensor Subset Formulation:} Rather than utilizing global network sensor counts (which lead to analytical underflows), we propose a localized critical-subset formulation ($m$) to model cyber-physical attacks realistic to smart grid operations.
    \item \textbf{Novel Comparative Static-Temporal Security Model:} We propose a first-order probabilistic security model that couples static cryptosystem compromise bounds with dynamic consensus-dependent exposure latency ($P_{secure}$).
    \item \textbf{Rigorous With-vs-Without Blockchain Evaluation:} We map and formulate the 12 cyber-physical attack success models explicitly under both centralized coordinator architectures and permissioned BFT blockchain environments.
    \item \textbf{Consensus Selection Roadmap:} We supply an operational decision roadmap by compiling theoretical message complexities and normalizing heterogeneous performance metrics (throughput, latency, and communication overhead).
\end{itemize}

\subsection{System Model}

We model an EV-integrated IEEE 33-bus distribution system (33 buses, 32 branches, peak load of 3715\,kW) with 50 EVs serving as prosumers. This yields $n=51$ participant validator nodes. The system contains $n_{sen} = 3854$ sensors:
\begin{equation}
n_{sen} = n_{sen}^{DN} (129) + n_{sen}^{EV} (3725) = 3854
\end{equation}

\subsection{Sensor Count Dominance Critique}

Prior models of smart meter security express sensor compromise probability ($P_{SA}$) and communication link compromise probability ($P_{CA}$) as:
\begin{equation}
P_{SA} = P_{CA} = x^{n_{sen}}
\end{equation}
where $x \in [0.9, 0.999]$ is the per-sensor security level. At $x = 0.95$ and $n_{sen}=3854$, $x^{3854} \approx 10^{-86}$. This causes the sensor and communication terms to underflow to zero, making the system security appear entirely independent of sensor vulnerability.

In reality, an attacker only needs to compromise a localized subset of critical sensors ($m$) at key bus nodes (e.g., feeder heads or lateral branch points) to inject false data or disrupt charging operations. We define $m = 10$, restoring model sensitivity:
\begin{equation}
P_{SA}^{(10)} = P_{CA}^{(10)} = x^{10} = 0.95^{10} \approx 0.599
\end{equation}

\subsection{Justification and Causal Analysis of the 12 Cyber-Physical Attacks}

Without blockchain, the system relies on centralized coordinators or weak client-server protocols. The baseline success probabilities ($P_{atk}^j$) for 12 cyber-attacks are justified below:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Sensor Compromise ($P_{SA}$):} The probability of altering physical measurements on $m$ critical sensors:
    \begin{equation}
    P_{SA} = x^m
    \end{equation}
    *Causal mechanism:* The attacker gains physical access to $m$ local smart meters, altering voltage/current sensing calibrations. Under a client-server architecture, these values are accepted without cross-verification.
    *Baseline parameter justification:* We use a baseline sensor security value $x = 0.95$ representing standard hardware reliability levels without tamper-resistant modules. To evaluate the robustness of this parameter, we perform a sensitivity analysis across $x \in [0.90, 0.999]$ in Section~\ref{sec:verification}.

    \item \textbf{False Data Injection ($P_{FDI}$):} Successful modification of meter readings submitted to state estimation:
    \begin{equation}
    P_{FDI} = x^m
    \end{equation}
    *Causal mechanism:* Altered meter values are transmitted to the coordinator. The system updates the state estimate using the falsified measurements, resulting in over-billing or grid frequency imbalances.
    *Baseline parameter justification:* Mirrored from sensor compromise.

    \item \textbf{Communication Compromise ($P_{CA}$):} Intercepting and altering data over transmission channels. Without end-to-end cryptographic integrity:
    \begin{equation}
    P_{CA} = x^m
    \end{equation}
    *Causal mechanism:* Data is modified in-transit across unencrypted cellular/Ethernet links linking smart meters to the central database.
    *Baseline parameter justification:* Mirrored from sensor compromise.

    \item \textbf{Man-in-the-Middle ($P_{MitM}$):} Intercepting session traffic. Without blockchain signatures, this is governed by session vulnerability $y$ (e.g., TLS session hijacking or DNS spoofing):
    \begin{equation}
    P_{MitM} = y \quad (y = 0.05)
    \end{equation}
    *Causal mechanism:* The attacker intercepts communication between an EV and the coordinator, injecting fake transaction requests while masquerading as the target node.
    *Baseline parameter justification:* The value $y = 0.05$ is derived from empirical network security studies on legacy smart grid protocols (e.g., Modbus/TCP or DNP3 without secure authentication) under active sniffing environments~\cite{baumeister2010}.

    \item \textbf{Replay Attack ($P_{Replay}$):} Resending captured energy transactions. Without sequential nonces or block-specific timestamp verification, the success rate is $z$:
    \begin{equation}
    P_{Replay} = z \quad (z = 0.15)
    \end{equation}
    *Causal mechanism:* An attacker intercepts a signed energy transfer confirmation and replays the packet, triggering duplicate credits or debits in the central billing database.
    *Baseline parameter justification:* The value $z = 0.15$ aligns with documented statistics on packet injection success rates in legacy, stateless smart meter nodes~\cite{yan2012}.

    \item \textbf{Sybil Attack ($P_{Sybil}$):} Forging identities to manipulate trading queues. Without validator admission control, registration is open:
    \begin{equation}
    P_{Sybil} = 1.0
    \end{equation}
    *Causal mechanism:* An attacker creates thousands of virtual nodes to dominate the bidding pool, crowding out legitimate EV traders.
    *Baseline parameter justification:* Open networks with simple IP-based registration have no defenses against virtual identity generation, making success guaranteed ($P=1.0$).

    \item \textbf{Denial of Service ($P_{DoS}$):} Flood attack targeting the central coordinator node. Governed by single-node resource exhaustion probability $p_{dos}$:
    \begin{equation}
    P_{DoS} = p_{dos} \quad (p_{dos} = 0.20)
    \end{equation}
    *Causal mechanism:* A single attacker floods the centralized coordinator's network port, causing it to drop energy matching requests.
    *Baseline parameter justification:* Based on standard single-server port-flooding vulnerability metrics under moderate firewall configurations~\cite{wang2013}.

    \item \textbf{Distributed DoS ($P_{DDoS}$):} Coordinated flood attack:
    \begin{equation}
    P_{DDoS} = p_{ddos} \quad (p_{ddos} = 0.35)
    \end{equation}
    *Causal mechanism:* A botnet saturates the coordinator's ingress bandwidth, cutting off all EVs from the energy market.
    *Baseline parameter justification:* Standard botnet traffic saturation probabilities on single-node public IP endpoints without dedicated DDoS mitigation service~\cite{wang2013}.

    \item \textbf{Byzantine Validator Attack ($P_{Byz}$):} Success rate if the central coordinator node fails or acts maliciously:
    \begin{equation}
    P_{Byz} = 1.0
    \end{equation}
    *Causal mechanism:* Because the coordinator holds absolute authority, any database error, insider threat, or software bug immediately compromises the entire system.
    *Baseline parameter justification:* Centralized architectures have a single point of failure (SPOF) with compromise probability $1.0$ if the core coordinator goes Byzantine.

    \item \textbf{Key Compromise ($P_{Key}$):} Theft of central coordinator signing keys:
    \begin{equation}
    P_{Key} = p_{key} \quad (p_{key} = 0.01)
    \end{equation}
    *Causal mechanism:* The private key is extracted from the coordinator's filesystem via malware, allowing the attacker to sign arbitrary transactions.
    *Baseline parameter justification:* Aligns with standard hardware security module (HSM) bypass or server key file leakage probabilities in corporate intranets~\cite{sridhar2012}.

    \item \textbf{SCADA Compromise ($P_{SCADA}$):} Exploiting SCADA control databases:
    \begin{equation}
    P_{SCADA} = 0.01
    \end{equation}
    *Causal mechanism:* Vulnerable database access interfaces permit SQL injection or credential theft at the SCADA server.
    *Baseline parameter justification:* Derived from Sheikh et al.'s SCADA vulnerability baseline ($P_{SCADA} = 0.01$)~\cite{sheikh2020}.

    \item \textbf{Receiver Compromise ($P_R$):} Overriding actuator command interfaces:
    \begin{equation}
    P_R = 0.01
    \end{equation}
    *Causal mechanism:* Actuators (switches, relays) are compromised via firmware exploits or default password access, overriding coordinator commands.
    *Baseline parameter justification:* Derived from Sheikh et al.'s receiver vulnerability baseline ($P_{R} = 0.01$)~\cite{sheikh2020}.
\end{enumerate}

\subsection{Total Probability of Security Without Blockchain}

Assuming these attacks represent parallel failure modes, the system remains secure only if \emph{all} attack vectors fail. The baseline probability of security ($P_{secure}$) without blockchain protection is:
\begin{equation}
P_{secure}^{no\_BC} = \prod_{j=1}^{12} (1 - P_{atk}^j)
\end{equation}
The proposed 12-attack model is intended as a conservative upper-bound approximation rather than an exact stochastic representation of correlated cyber-physical attacks. Although cyber-physical attacks can be correlated in practice, this independence assumption enables closed-form probability calculations to establish a baseline worst-case exposure.

Using $m=10$, $x=0.95$, $y=0.05$, $z=0.15$, $p_{dos}=0.20$, $p_{ddos}=0.35$, $p_{key}=0.01$, and SCADA/receiver $P=0.01$, the probability of security without blockchain evaluates to:
\begin{equation}
\begin{split}
P_{secure}^{no\_BC} &= (1-0.599)^3(1-0.05)(1-0.15)(1-1)\\
&\quad \times (1-0.2)(1-0.35)(1-1)(1-0.01)^3
\end{split}
\end{equation}
Since $P_{Sybil} = 1.0$ and $P_{Byz} = 1.0$ under centralized architectures, these single points of failure drive the overall security to absolute zero, yielding $P_{secure}^{no\_BC} = 0.0$.

To establish a conservative and fair baseline comparison that isolates cyber-physical attacks, we temporarily exclude these catastrophic single-points-of-failure:
\begin{equation}
\begin{split}
P_{secure, limited}^{no\_BC} &= (1-0.599)^3(1-0.05)(1-0.15)\\
&\quad \times (1-0.2)(1-0.35)(1-0.01)^3 \\
&\approx 0.026
\end{split}
\end{equation}
This baseline shows the extreme vulnerability of centralized EV networks under the modeled threat parameters. Mathematically, a series reliability model dictates that the system is only secure if \emph{every single} vulnerability is defended simultaneously. Because probabilities are multiplicative (e.g., $(1 - 0.599)^3 \times (1 - 0.20)$ etc.), the cumulative chance of surviving all 12 attack vectors decays exponentially. 

As calculated above, a value of $0.026$ indicates that \textbf{the system is only 2.6\% safe without blockchain}. Conversely, this means there is a massive \textbf{97.4\% probability that the centralized baseline will be successfully compromised} by at least one of these vectors. This establishes a highly vulnerable baseline. To validate these assumptions against arbitrary changes in baseline parameters, we perform sensitivity sweeps in Section~\ref{sec:verification}.

\section{Probability Analysis: Expected Value vs. System Compromise}
\label{sec:prob_analysis}

A critical distinction must be made between the baseline established by Sheikh et al.~\cite{sheikh2020} and the parallel failure model extended in this work.

Sheikh et al.\ evaluated the \textbf{Target Attack Probability ($P_{TA}$)}. Their model assumes an attacker randomly targets one of four subsystems (Sensor, Communication, SCADA, or Receiver) with uniform probability ($1/4$). Thus, $P_{TA}$ is the arithmetic mean, yielding the expected value of success for a single, randomly targeted attack:
\begin{equation}
P_{TA} = \frac{1}{4}\left(2P_{SA} + P_{SCADA} + P_R\right) \approx 0.005
\end{equation}

In contrast, our proposed 12-attack extension models \textbf{System Compromise Probability ($P_{Compromise}$)}. It assumes a worst-case parallel exposure where an attacker launches all attacks simultaneously, and the system fails if \emph{any} single attack vector succeeds (a series reliability model):
\begin{equation}
P_{Compromise} = 1 - \prod_{j=1}^{12} (1 - P_{atk}^j) \approx 0.974
\end{equation}

The original paper is mathematically correct for its defined scope (expected value of a random target attack). Our extension demonstrates that when evaluated as a parallel system compromise, the baseline centralized vulnerability is significantly higher (0.974 vs 0.005). We use Sheikh's $P_{TA}$ for temporal benchmarking to maintain comparative validity with the original work, while using $P_{Compromise}$ to illustrate the broader structural vulnerability of centralized grids.

\subsection{Causal Chain of the 12 Cyber-Physical Attacks}
\label{sec:twelve_attacks}

We now list the detailed formulations for each of the 12 attack vectors.

\begin{enumerate}[leftmargin=*]
    \item \textbf{Sensor ($P_{\mathrm{Sensor}}$) \& FDI ($P_{\mathrm{FDI}}$):} \emph{[Proposed]} Nodes cross-verify block data with cryptographic hashes. Compromising these requires compromising both the sensor physical layer and private key storage:
    \begin{equation}
    \label{eq:sensor_fdi_bc}
    P_{\mathrm{Sensor}} = P_{\mathrm{FDI}} = x^{2m}
    \end{equation}
    \paragraph{Causal chain.} The attacker attempts to inject false meter data. P2P validator nodes detect the mismatch between hashed records and physical signatures, dropping the block. The attack fails unless the attacker controls the private keys of the sensors to generate matching signatures~\cite{ding2024}.
    \paragraph{Defense justification.} Unlike centralized systems where altered measurements are immediately accepted, BFT consensus requires individual cryptographic verification. The attacker must execute two independent breaches for each smart meter in the critical subset: physically override the sensing hardware (probability $x^m$) and extract the matching local private key from secure storage to generate valid signatures (probability $x^m$), resulting in the joint product $x^m \cdot x^m = x^{2m}$.

    \item \textbf{Communication ($P_{\mathrm{Comm}}$):} \emph{[Proposed]} The attacker must manipulate consensus messages across P2P validator links:
    \begin{equation}
    \label{eq:comm_bc}
    P_{\mathrm{Comm}} = x^{2k_1}, \quad k_1 = \frac{m(m-1)}{2} = 45
    \end{equation}
    \paragraph{Causal chain.} Data in transit is signed. Modification of packets triggers validation failure during the prepare phase~\cite{castro1999}.
    \paragraph{Defense justification.} In a decentralized configuration, $m$ local nodes must reach consensus, forming a fully mesh-connected local network with $k_1 = \frac{m(m-1)}{2}$ directional links. Since transmissions are signed, hijacking requires both routing-layer interception (probability $x^{k_1}$) and cryptographic session key extraction (probability $x^{k_1}$), producing $x^{2k_1}$.

    \item \textbf{MitM ($P_{\mathrm{MitM}}$):} \emph{[Proposed]} Transaction signing requires compromising sender keys, squaring vulnerability:
    \begin{equation}
    \label{eq:mitm_bc}
    P_{\mathrm{MitM}} = y^2 = 0.0025
    \end{equation}
    \paragraph{Causal chain.} Intercepting session traffic is insufficient. Without the sender's private key, the attacker cannot inject valid transaction payloads~\cite{baumeister2010}.
    \paragraph{Defense justification.} In a stateless blockchain system, session hijacking (probability $y$) does not grant write access. To alter or inject transactions, the attacker must additionally extract the private key of the EV trader (modeled by baseline probability $y$), resulting in the squared probability $y^2$.

    \item \textbf{Replay ($P_{\mathrm{Replay}}$):} \emph{[Adopted]} Cryptographic block sequencing, nonces, and timestamps reduce replay success to zero:
    \begin{equation}
    \label{eq:replay_bc}
    P_{\mathrm{Replay}} \approx 0.0
    \end{equation}
    \paragraph{Causal chain.} Replayed transactions contain stale nonces or timestamps, which are rejected during validator pre-prepare checks~\cite{yan2012}.

    \item \textbf{Sybil ($P_{\mathrm{Sybil}}$):} \emph{[Adopted]} Admission control limits the validator pool ($n=51$). An attacker must inject $i > f$ malicious validator nodes:
    \begin{equation}
    \label{eq:sybil_bc}
    P_{\mathrm{Sybil}} = \sum_{i=f+1}^{n} \binom{n}{i} p_c^i (1 - p_c)^{n-i} \approx 1.54 \times 10^{-10}
    \end{equation}
    where $p_c = 0.05$ is the probability of a validator being compromised.
    \paragraph{Causal chain.} Because the validator pool is permissioned, the attacker cannot spin up arbitrary node identities. They must hack existing validator nodes~\cite{li2020_blockchain}.

    \item \textbf{DoS ($P_{\mathrm{DoS}}$) \& DDoS ($P_{\mathrm{DDoS}}$):} \emph{[Modified]} Consensus continues as long as $\geq n - f$ nodes are online. The attacker must offline at least $f+1$ nodes:
    \begin{equation}
    \label{eq:dos_bc}
    P_{\mathrm{DoS}} = p_{dos} \cdot \left(\frac{f+1}{n}\right) \approx 0.067
    \end{equation}
    \begin{equation}
    \label{eq:ddos_bc}
    P_{\mathrm{DDoS}} = p_{ddos} \cdot \left(\frac{f+1}{n}\right) \approx 0.117
    \end{equation}
    \paragraph{Causal chain.} Flooding one or a few nodes does not halt trading. The P2P network routes around offlined nodes, maintaining consensus~\cite{wang2013}.
    \paragraph{Defense justification.} A BFT consensus ledger can only be stalled if at least $f+1$ nodes are compromised or flooded. The linear scale factor $\frac{f+1}{n}$ models the probability of random target selection hitting the minimum threshold of nodes required to stall consensus under a uniform target distribution.

    \item \textbf{Byzantine Validator Attack ($P_{\mathrm{Byz}}$):} \emph{[Adopted]} BFT consensus guarantees correctness if malicious nodes do not exceed $f=16$:
    \begin{equation}
    \label{eq:byz_bc}
    P_{\mathrm{Byz}} = \sum_{i=f+1}^{n} \binom{n}{i} p_{byz}^i (1 - p_{byz})^{n-i} \approx 1.54 \times 10^{-10}
    \end{equation}
    \paragraph{Causal chain.} No single validator node has absolute authority. Malicious proposal changes are voted down~\cite{castro1999}.

    \item \textbf{Key Compromise ($P_{\mathrm{Key}}$):} \emph{[Adopted]} Central coordinator signing keys are eliminated. Validator keys are secured using Shamir $(3,5)$ secret sharing:
    \begin{equation}
    \label{eq:key_bc}
    P_{\mathrm{Key}} = P_{R,shamir}(3,5) \approx 9.85 \times 10^{-6}
    \end{equation}
    \paragraph{Causal chain.} Key shares are distributed across independent execution environments. The attacker must compromise at least three shares simultaneously to sign blocks~\cite{sridhar2012}.

    \item \textbf{SCADA ($P_{\mathrm{SCADA}}$) \& Receiver ($P_{\mathrm{Receiver}}$):} \emph{[Modified]}
    \begin{equation}
    \label{eq:scada_bc}
    P_{\mathrm{SCADA}} = (P_{SCADA} \cdot x)^m = (0.01 \cdot 0.95)^{10} \approx 5.99 \times 10^{-21}
    \end{equation}
    \begin{equation}
    \label{eq:receiver_bc}
    P_{\mathrm{Receiver}} = P_R^{k_2} \cdot x^m, \quad k_2 = \lfloor m \times 0.33 \rfloor = 3 \implies 5.99 \times 10^{-7}
    \end{equation}
    \paragraph{Causal chain.} SCADA commands must match blockchain state, and receivers execute commands only after consensus confirmation~\cite{sheikh2020}.
    \paragraph{Defense justification.} Commands sent to RTUs or actuators are cross-referenced with the blockchain state ledger. Compromising the receiver requires physically overriding the actuator ($P_R^{k_2}$) AND compromising the validating sensor nodes ($x^m$). Since BFT requires committee signatures, the receiver checks for $k_2 = \lfloor m \times 0.33 \rfloor$ matching validator signatures before execution.
\end{enumerate}

\subsection{Static Security Verdict}
\label{sec:static_verdict}

The proposed probabilistic extension demonstrates that blockchain consensus eliminates the two most critical single-points-of-failure ($P_{\mathrm{Sybil}}$ and $P_{\mathrm{Byz}}$ reduced from 1.0 to $\approx 10^{-10}$), disables replay attacks ($P_{\mathrm{Replay}} \approx 0$), and significantly reduces DoS/DDoS, MitM, and key compromise probabilities. 

\emph{Bridging the Models:} The dominant residual blockchain attack vector is the sensor compromise $P_{\mathrm{Sensor}} = x^{2m} \approx 0.358$, reflecting that physical sensor tampering remains a challenge even with blockchain. Readers must note the distinction between the two evaluation models applied here: the four-component model is used for the static benchmark ($10^{170}$ improvement) because it isolates cyber-layer vulnerabilities. Conversely, the twelve-attack model is used to analyze comprehensive attack coverage, demonstrating that while blockchain eliminates cyber-layer failures (like Sybil and Key Compromise), physical-layer compromises (like localized sensor attacks) mathematically dominate the residual risk.

For the primary quantitative comparison, we use the Sheikh et al.\ four-component model (Section~\ref{sec:sheikh_comparison}), which yields:
\begin{equation}
\text{Static Security Gain} = \frac{P_{TA}}{P_{TAb}} \approx \frac{0.005}{4.91 \times 10^{-173}} \approx 10^{170}
\end{equation}
Deploying blockchain consensus reduces the modeled static attack probability by approximately \textbf{170 orders of magnitude} under the Sheikh four-component probabilistic formulation. This confirms the central thesis of Sheikh et al.~\cite{sheikh2020}: blockchain fundamentally improves application security.

\subsection{Per-Algorithm Probability of Compromise}
\label{sec:per_algo_compromise}

Having established in Table~\ref{tab:comparison} that blockchain reduces all 12 attack probabilities, we now evaluate how individual consensus protocols affect each attack vector. 

To obtain distinct, mathematically justified probabilities across protocols, we model each attack probability $P_j^{\text{Algo}}$ as a function of the protocol's architectural properties (committee size $n_{val}$, fault threshold $f$, voting phases $N_{phases}$, message complexity $M_c$, latency $L$, credit weighting, and VRF leader election):
\begin{enumerate}[leftmargin=*]
    \item \textbf{Sensor ($P_{Sensor}^{\text{Algo}}$):} Larger committees provide greater validator redundancy and spatial cross-validation check coverage, reducing the probability of sensor tampering:
    \begin{equation}
    P_{Sensor}^{\text{Algo}} = x^{2m} \cdot \left(1 - \eta \cdot \frac{n_{val}}{n_{global}}\right)
    \end{equation}
    where $\eta = 0.15$ is the spatial cross-verification factor and $n_{global}=51$ is the total validator network size.
    
    \item \textbf{False Data Injection ($P_{FDI}^{\text{Algo}}$):} Protocols with more voting phases perform sequential multi-round voting, increasing the likelihood of filtering malicious state changes before finality:
    \begin{equation}
    P_{FDI}^{\text{Algo}} = x^{2m} \cdot \left(1 - \beta \cdot N_{phases}\right)
    \end{equation}
    where $\beta = 0.10$ is the per-phase verification factor.
    
    \item \textbf{Communication Hijack ($P_{Comm}^{\text{Algo}}$) \& DoS/DDoS ($P_{DoS/DDoS}^{\text{Algo}}$):} High message complexity increases the active network attack surface and validator queue saturation, making the system more vulnerable to network jamming and session flooding:
    \begin{equation}
    P_{Comm}^{\text{Algo}} = x^{2k_1} \cdot \sigma_{CO}^{\text{Algo}}
    \end{equation}
    \begin{equation}
    P_{DoS}^{\text{Algo}} = p_{dos} \cdot \left(\frac{f+1}{n_{val}}\right) \cdot \sigma_{CO}^{\text{Algo}} \cdot \left(1 - e^{-L^{\text{Algo}} / \tau_{dos}}\right)
    \end{equation}
    where $\tau_{dos}=1.0$\,s, and the log-normalized communication overhead is:
    \begin{equation}
    \sigma_{CO}^{\text{Algo}} = \frac{\log_{10}(M_c^{\text{Algo}})}{\log_{10}(M_{c,OM(m)}^{\text{BC}})}
    \end{equation}
    
    \item \textbf{Man-in-the-Middle ($P_{MitM}^{\text{Algo}}$):} MitM spoofing requires forging signatures across the entire validating committee. Larger committees dilute the probability of successful group-key hijacking:
    \begin{equation}
    P_{MitM}^{\text{Algo}} = \frac{y^2}{n_{val}}
    \end{equation}
    
    \item \textbf{Sybil ($P_{Sybil}^{\text{Algo}}$) \& Byzantine ($P_{Byz}^{\text{Algo}}$):} These scale with the binomial tail of validator compromise:
    \begin{equation}
    P_{Sybil}^{\text{Algo}} = P_{Byz}^{\text{Algo}} = \sum_{i=f+1}^{n_{val}} \binom{n_{val}}{i} (p_{c,eff})^i (1 - p_{c,eff})^{n_{val}-i}
    \end{equation}
    where reputation-weighted credit-evaluation reduces the effective compromise probability to $p_{c,eff} = p_c(1 - \omega_{credit}) = 0.03$ (for CE-PBFT, SV-PBFT, and RVR, where $\omega_{credit}=0.40$), while others use $p_c = 0.05$.
    
    \item \textbf{SCADA Breach ($P_{SCADA}^{\text{Algo}}$) \& Key Compromise ($P_{Key}^{\text{Algo}}$):} These rely on validator secret key integrity, which scales down under credit-based validator selection:
    \begin{equation}
    P_{Key}^{\text{Algo}} = P_{R,shamir}(3, 5, p_{r,eff\_key})
    \end{equation}
    \begin{equation}
    P_{SCADA}^{\text{Algo}} = (P_{SCADA} \cdot x)^m \cdot p_{c,eff} \cdot \left(1 - e^{-L^{\text{Algo}} / \tau_{scada}}\right)
    \end{equation}
    where $p_{r,eff\_key} = P_R(1 - \omega_{credit})$ ($\omega_{credit}=0.40$ for credit-evaluated, $0.0$ otherwise), and $\tau_{scada}=5.0$\,s.
    
    \item \textbf{Receiver Actuator Override ($P_{Receiver}^{\text{Algo}}$):} VRF-based proposer hiding and fast finality minimize the attacker's window of opportunity:
    \begin{equation}
    P_{Receiver}^{\text{Algo}} = (p_{r,eff\_recv})^{k_2} \cdot x^m \cdot \left(1 - e^{-L^{\text{Algo}} / \tau_{recv}}\right)
    \end{equation}
    where $\tau_{recv}=2.0$\,s, and $p_{r,eff\_recv} = P_R / n_{val}$ for VRF-enabled protocols (RVR), else $P_R = 0.01$.
\end{enumerate}

\subsection{Independent Failure Assumption and Modeling Limitations}
\label{sec:independence_limitations}

We acknowledge that the joint compromise probability $P_{\mathrm{Compromise}} = 1 - \prod_{j} (1 - P_j)$ assumes that all 12 attack vectors act as independent, parallel failure modes. In dynamic industrial networks, cyber-physical attacks are often correlated; for example, a successful SCADA breach significantly facilitates key compromise and actuator overrides. Our independent failure model represents a conservative, structural upper-bound failure limit for comparison. Future research will employ Bayesian Belief Networks (BBN) or Markov Chains to model sequential state transition probabilities and conditional failure cascades across physical smart meters, consensus protocols, and RTU actuators.

\section*{Part II: Consensus Evaluation}
\section{Temporal Vulnerability}
\label{sec:temporal_benchmarking}

\emph{Having established that blockchain improves static security by $\sim$170 orders of magnitude (Part~I), we now evaluate how the temporal exposure window of different consensus algorithms affects the realized security gains.}

\subsection{Temporal Exposure Window}

Although blockchain provides static security, transactions remain unfinalized during the consensus process. This latency window ($L$) exposes the network to temporal attacks. We model this exposure using a Poisson attack arrival process with rate $\lambda$ (attacks/second):
\begin{equation}
P_{temporal}(\lambda, L) = 1 - e^{-\lambda \cdot L \cdot P_{TA}}
\end{equation}

The overall security probability ($P_{secure}$) incorporates this temporal risk:
\begin{equation}
P_{secure} = (1 - P_{TAb})(1 - P_{temporal}(\lambda, L))(1 - P_{other})
\end{equation}
where $P_{other} = 0.05$ represents residual risks.


The following protocol comparison assumes blockchain has already been selected. The objective is no longer to determine whether blockchain should be deployed, but rather which consensus protocol provides the best operational trade-off.

\section{Protocol Comparisons}

\subsection{Shortlisted Architectural Security and Performance Features}

We focus on four features that directly influence system metrics:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Proof of History (PoH):} VDF-based clock. Eliminates synchronous communication rounds, reducing message overhead from $\mathcal{O}(n^2)$ to $\mathcal{O}(n)$, yielding sub-second latency ($L=243$\,ms) and restoring $P_{secure}$ to 0.927~\cite{shoup2023}. 
\textbf{Tower BFT Analytical Latency Derivation:} The 242.9\,ms latency evaluated in Table~\ref{tab:security} represents an empirical measurement. Analytically, Tower BFT latency $L$ is bounded by the network flight time ($\tau=50$\,ms), PoH tick verification ($\sim42.9$\,ms), and turbine tree propagation ($\sim37.1$\,ms for $n=51$). Thus, the theoretical lower bound is $L_{base} = 2\tau + L_{PoH} + L_{turbine} \approx 180$\,ms. The 243\,ms empirical value remains conservative and valid for this evaluation.

    \item \textbf{VRF Leader Election:} Random verifiable leader rotation. Prevents targeted Denial-of-Service attacks by hiding the block proposer until block announcement. Reduces effective receiver compromise probability from $P_R=0.01$ to $P_{R,VRF}=P_R/n \approx 1.96 \times 10^{-4}$~\cite{wang2025}.
    \item \textbf{Geohash Geographic Clustering:} Reduces communication complexity from global validator network size $n$ to localized cluster size $c$. Reduces PBFT message counts from 7,854 to 1,900~\cite{xu2025}.
    \item \textbf{Credit-Evaluation Scoring:} Dynamically adjusts validator weights based on behavior history, reducing the threat probability of malicious validators ($p_c$)~\cite{ding2024}.
\end{enumerate}

\section{Key Management and Defense-in-Depth}
\label{sec:key_management}
% ═══════════════════════════════════════════════════════════════

To secure validator node credentials, we evaluate three threshold key management schemes:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Shamir Secret Sharing $(k,d)$:} Reconstructing the key requires $k$ out of $d$ shares. The compromise probability is the tail of a binomial distribution:
    \begin{equation}
    P_{R,shamir}(k,d) = \sum_{i=k}^{d} \binom{d}{i} P_{share}^i (1 - P_{share})^{d-i}
    \end{equation}
    With $P_{share} = 0.01$, a $(3,5)$ configuration yields $P_{R,shamir} \approx 9.85 \times 10^{-6}$, representing a $1,015\times$ risk reduction.

    \item \textbf{Multi-Party Computation (MPC):} Eliminates share reconstruction by calculating signatures distributively. Round-based isolation reduces leakage by factor $\eta_{MPC} = 0.7$:
    \begin{equation}
    P_{R,MPC}(k,n) = 0.7 \cdot P_{R,shamir}(k,n) \implies 6.90 \times 10^{-6} \text{ for } (3,5)
    \end{equation}

    \item \textbf{Multisig Thresholds:} Validators sign using independent keys, with independence factor $\eta_{sig} = 0.85$:
    \begin{equation}
    P_{R,multisig}(k,n) = 0.85 \cdot P_{R,shamir}(k,n) \implies 8.37 \times 10^{-6} \text{ for } (3,5)
    \end{equation}
\end{enumerate}

These key management schemes provide defense-in-depth, reducing the validator key compromise risk that underpins $P_{Rb}$ and SCADA vulnerabilities.

\subsection{The Latency-Security Trade-Off Pareto Frontier}

Since $P_{TAb} \approx 0.0$ for all blockchain configurations, the overall security probability is determined by the temporal vulnerability:
\begin{equation}
P_{secure} \propto e^{-L}
\end{equation}
Lower latency directly translates to higher overall security. However, lower latency often requires smaller committees or relaxed consensus constraints, which can impact decentralization.

\subsection{Consensus Selection Roadmap}

Based on our analysis, we propose the following consensus selection matrix for smart grid operators:

\begin{itemize}[leftmargin=*]
    \item **High-Security Environments (e.g., transmission control):** RVR or Tower BFT ($P_{secure} \geq 0.927$, sub-second latency).
    \item **High-Throughput Microgrids (e.g., dense local EV trading):** CE-PBFT or SV-PBFT (balance of >60 TPS throughput and >0.87 security).
    \item **Enterprise Consortiums (e.g., regional utility settlement):** QBFT (low message overhead, 1.5s block finality).
\end{itemize}

\subsection{Threats to Validity}

To present a balanced and mature scientific evaluation, we identify the following limitations and threats to the validity of our findings:
\begin{enumerate}[leftmargin=*]
    \item \textbf{Construct Validity (Base Probability Parameters):} The baseline probability constants ($y=0.05, z=0.15$) represent assumed operational points. While justified by literature, in real deployments, these fluctuate based on network configurations. We mitigate this threat by conducting sensitivity sweeps over the parameters.
    \item \textbf{Internal Validity (Attack Independence):} Assuming independent Bernoulli trials ignores multi-stage attack correlations. While this standard analytical approach allows closed-form comparisons, it remains a threat to model accuracy, which we intend to address using copula modeling.
    \item \textbf{External Validity (IEEE 33-Bus Generalizability):} Smart grid topologies vary widely. Our parameters are optimized for a medium-scale distribution feeder (51 nodes) and may scale non-linearly on transmission networks with thousands of buses.
    \item \textbf{Experimental Validity (Heterogeneous Testbeds):} Performance parameters for Groups 2--4 are compiled from different experimental publications (e.g., FISCO BCOS, Hyperledger Fabric, and simulator environments). Direct quantitative comparison carries inherent environments bias, which we mitigate by analyzing normalized relative speedup factors.
\end{enumerate}

\section{Conclusion}
\label{sec:conclusion}
% ═══════════════════════════════════════════════════════════════

\subsection{Conclusion A: Security Gain of Blockchain Deployment}

Deploying BFT blockchain consensus on V2G energy trading systems resolves central coordinator vulnerabilities and mitigates Sybil and replay exploits. Using the foundational four-component attack model of Sheikh et al.~\cite{sheikh2020}, the static attack probability drops from $P_{TA} \approx 0.005$ to $P_{TAb} \approx 10^{-173}$---a static security improvement of approximately \textbf{170 orders of magnitude} under the Sheikh four-component probabilistic formulation. The complementary proposed probabilistic extension confirms that blockchain eliminates the single-points-of-failure ($P_{Sybil}, P_{Byz}$: $1.0 \to 10^{-10}$) and neutralizes replay attacks.

\subsection{Conclusion B: Protocol Selection Recommendations}

While blockchain significantly improves static security, the consensus latency window introduces temporal vulnerability. Under an FDI threat profile of $\lambda=20$\,s$^{-1}$, protocols with higher latency (e.g., OM($m$)) can lose up to 99\% of the static security gain ($P_{secure} = 0.012$). Pipelined sub-second protocols (e.g., RVR, Tower BFT) preserve over 92\% of the security benefits ($P_{secure} \geq 0.927$), making them the recommended choice for real-time smart grid applications.
